## 🔥 Top 7 Tips & Tricks
1. **Pin Actions to a full SHA** — `uses: actions/checkout@<40-char-sha>` instead of `@v4`/`@main`. Mutable tags are a live supply-chain attack vector; a compromised tag runs arbitrary code in your pipeline.
2. **Use OIDC, not static cloud keys** — Have GitHub Actions assume a cloud role via OIDC federation. No long-lived `AWS_SECRET_ACCESS_KEY` in secrets to leak or rotate.
3. **Multi-stage Docker builds + slim bases** — Build in a fat image, copy only the artifact into `node:20-alpine` / `python:3.12-slim`. Smaller image = smaller attack surface = faster pulls.
4. **Always declare HEALTHCHECK / probes** — Add `HEALTHCHECK` in the Dockerfile and map to Kubernetes `liveness`/`readiness` probes so orchestrators restart unhealthy containers instead of routing traffic to them.
5. **Set `unhealthyPodEvictionPolicy: AlwaysAllow` on PDBs** — Lets misbehaving pods be evicted during a drain instead of blocking it; the #1 cause of "kubectl drain hangs forever".
6. **Terraform provisions, Ansible configures — via dynamic inventory** — Keep them in one repo, separate dirs. Prefer Terraform outputs / cloud inventory plugins over `local-exec` provisioners so an Ansible failure can't corrupt Terraform state mid-apply.
7. **Right-size the AWS container platform** — ECS on Fargate ($0 control plane, ~$14–18/mo for a small service) under ~15 services; EKS + Karpenter + Spot once you're past ~15–20 always-on containers or need multi-cloud / many tenants.

## 🐛 Problems & Solutions

### Problem 1: `kubectl drain` hangs forever during node maintenance
- **Problem**: Draining a node for an upgrade stalls; pods won't evict and the operation never completes.
- **Root cause**: The `PodDisruptionBudget` defaults to waiting for application pods to become healthy before allowing eviction. If a pod is stuck unhealthy, or the PDB would drop below its `minAvailable`, the Eviction API denies the request indefinitely.
- **Fix**:
  - Set `unhealthyPodEvictionPolicy: AlwaysAllow` on the PDB so unhealthy pods can always be evicted.
  - Ensure the PDB permits **at least one** disruption (`minAvailable` < replica count, or `maxUnavailable: 1`).
  - Remember rolling updates are bounded by the Deployment's `maxUnavailable`/`maxSurge`, **not** the PDB — set both consistently.

### Problem 2: GitHub Actions pipeline compromised via a tag
- **Problem**: A third-party action you reference by `@v4` silently starts exfiltrating secrets.
- **Root cause**: Tags and branches are mutable. An attacker who controls the action repo can repoint `v4` to malicious code that runs with your workflow's permissions.
- **Fix**: Pin every action to a full commit SHA, apply least-privilege `permissions:` at the workflow/job level, and authenticate to the cloud with OIDC instead of stored credentials. Add Dependabot to bump the pinned SHAs safely.

## 🏗️ Architecture Pick of the Day

### ECS on Fargate vs EKS

| | ECS on Fargate | EKS |
|---|---|---|
| **Use case** | AWS-native teams, small-to-mid container fleets, fastest path to prod | Kubernetes ecosystem, multi-cloud/hybrid, many tenants, large scale |
| **Pros** | $0 control-plane fee; deploys in ~2 commands; zero K8s knowledge; ~$14–18/mo small svc | Full CNCF ecosystem; multi-cloud portable; namespace multi-tenancy; EKS Auto Mode + Karpenter + Spot for cost at scale |
| **Cons** | No K8s ecosystem; less portable; weaker fine-grained multi-tenancy | $0.10/cluster/hr (~$73/mo) before any workload; steeper learning curve; more ops surface |
| **Choose when** | New AWS project, < ~15 services, team lives in the AWS console | Concrete trigger: multi-cloud, internal platform w/ many tenants, K8s-native tooling dependency, or team already in `kubectl`; past ~15–20 always-on containers |

**Verdict**: For most teams starting on AWS in 2026, **ECS on Fargate wins by default** — adopt EKS deliberately when you have a concrete multi-cloud, multi-tenant, or scale trigger, at which point Karpenter + Spot makes the control-plane fee trivial.
